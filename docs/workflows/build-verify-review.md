# build-verify-review

## 进入条件

- 任务已进入实施阶段
- 需要补最小验证证据
- 需要把改动送入正式审查

## 实施前必须确认

正式动手前，至少确认：

- 当前边界是什么
- 当前动哪些文件
- 哪些旧功能可能受影响
- 当前计划是否还是最新
- 这次最小验证和回归验证怎么做

## 实施阶段控边界规则

- 只改当前切片需要的范围
- 不顺手做无关重构
- 多文件联动时先确认主文件和从文件
- 任务一旦变质，先停下更新计划

典型失控信号：

- 原本小改，开始跨多个模块
- 原本修 bug，开始重做命名或结构
- 原本改实现，顺手改协议、规则和目录体系

## 失败证据优先触发器

以下情况默认先有失败证据，再进入实现：

- bug 可复现
- 改公共模块逻辑
- 改接口或数据契约
- 修回归问题
- 改 AI 行为，例如 Prompt、模型、RAG、tool routing、安全策略

失败证据可以是：

- 失败测试
- 固定复现步骤
- 失败 eval case
- 已记录的回归场景

## 验证层次

`V3` 把验证分为 5 层：

1. 范围验证：这次改动有没有跑出原任务边界
2. 静态验证：lint、typecheck、build、基础检查
3. 功能验证：新功能能否使用，原问题是否修复
4. 回归验证：相关旧功能有没有被带坏
5. AI 行为验证：baseline、关键 case、回归 case、必要的 latency/cost/safety 影响

不是每次都要跑到最重，但不能跳过验证就宣布完成。

## 审查闭环

正式收口必须走：

`implement -> verify -> review -> fix -> verify -> review`

规则固定如下：

- review 不是走过场
- fix 之后不能直接结束
- 每次修正都要重新进入验证和 review

## 关键步骤

- 按计划或当前边界实施改动
- 运行最小必要验证并记录证据
- 按 `runtime/code_review.md` 做结构化审查
- 命中问题时进入 `fix -> verify -> review` 回环
- 收口前明确未验证项和剩余风险

## 出口产物

- 实施结果
- `Verification Evidence`
- `Accept / Hold / Reject` 结论
- 是否继续进入 `release-retro`
