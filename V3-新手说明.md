# V3 新手说明

## 文件定位

这份文档是给第一次接触 `V3` 的人看的导读。  
它负责把目录树、主流程和 `5` 个核心 skill 讲明白，但不替代正式真相源。

如果这里和正式规则有冲突，以这些文件为准：

1. `README.md`
2. `AGENTS.md`
3. `PROJECT_RULES.md`
4. `docs/`
5. `runtime/`

## 先用一句话认识 V3

`V3` 不是“全自动 agent 平台”，也不是“只给一个空目录树”的模板。  
它更像一套面向 Codex 的轻量工程闭环，核心目标是让任务从接入、实施、验证、审查到发布，都有清楚的入口、清楚的边界和清楚的留痕。

你可以把它先记成三层：

- `Core`：默认最小闭环，所有项目先靠这一层跑起来
- `Gates`：命中条件后必须加上的正式门禁
- `Packs`：只有项目真的需要时才启用的扩展层

## 第一次阅读建议

如果你是第一次看 `V3`，推荐按这个顺序读：

1. `README.md`
2. `AGENTS.md`
3. `PROJECT_RULES.md`
4. 本文
5. `docs/README.md`
6. `docs/workflows/`
7. `.agents/skills/`
8. `docs/protocols/`
9. `runtime/README.md`

这样读的好处是：先知道它是什么，再知道规则，再知道目录和流程，最后才进入具体 skill 和运行对象。

## 根级入口层在做什么

根级不是拿来堆很多运行对象的，它主要负责告诉你“先看哪里、再去哪儿”。

### `README.md`

这是产品级入口。  
它回答 4 个最基础的问题：

1. `V3` 是什么
2. `V3` 不是什么
3. 它适合什么项目
4. 它的总结构怎么分层

如果你还不知道这个模板是不是适合你的项目，先看这里。

### `AGENTS.md`

这是执行规则入口。  
它不是介绍目录，而是告诉 AI 或协作者：

- 什么时候必须先计划
- 什么时候不能直接结束
- 什么时候要补验证证据
- 什么时候要把内容沉淀进 `runtime/`

简单说，`README.md` 解决“这是什么”，`AGENTS.md` 解决“做事不能踩哪些线”。

### `PROJECT_RULES.md`

这是地图入口。  
它不展开详细流程，但会告诉你：

- 改什么东西先看哪里
- 哪些目录分别管什么
- 哪些改动会联动检查哪些地方

如果你已经知道要改什么，但不知道应该从哪个目录开始看，先看这个文件。

### `V3-新手说明.md`

也就是你现在正在看的这份。  
它是“给人解释”的导读文件，作用是把结构和流程说成人话。

### `V3-检查清单.md`

这是配套检查清单。  
当你读完整个 `V3` 后，可以拿它逐项核对有没有漏掉哪一层。

## `.agents/skills/` 是什么

这里放的是 `V3` 的核心 skill 包。  
你可以把每个 skill 当成一个“专门处理某一段任务的工作角色”。

`V3` 采用的是 `skill package first` 思路，也就是：

- 先按 skill 分目录
- skill 自己拥有自己的模板
- skill 自己拥有自己的私有脚本
- 单个 skill 的私有资源尽量不要散落到全局

这意味着你看一个 skill 时，通常会在同一个目录里看到：

- `SKILL.md`：这个 skill 的规则正文
- `templates/`：它自己专用的模板
- `scripts/`：它自己专用的辅助脚本

注意：这里我们只讲“什么时候触发、负责什么”，不展开正文细节。

### `dev-planner`

它负责“要不要计划，以及计划写到什么程度”。

典型触发场景：

- 任务还没动手，需要先判断边界
- 任务是中改、大改或高风险
- 任务会跨会话继续
- 任务会暂停后再恢复
- 任务需要正式交接给别人接着做
- 原计划已经失效，需要更新

它的核心作用不是把所有任务都写成长计划，而是先做分流：

- `极小`：边界很清楚，直接给边界和最小验证
- `小`：给一份轻计划
- `中`：优先走官方计划能力，再判断要不要沉淀到仓库
- `大`：必须认真规划，必要时沉淀到 `runtime/PLANS.md`

一句人话：  
`dev-planner` 不是为了“把计划写长”，而是为了“让任务不要一上来就做乱”。

### `dev-builder`

它负责“按边界实施改动”。

典型触发场景：

- 当前任务边界已经明确
- 已经有轻计划或正式计划
- 要开始真正改代码、改配置或改正式实现
- review 提了问题，需要进入修后回环

它的核心职责有 3 个：

1. 按当前边界实施
2. 补最小必要验证证据
3. 把结果送去 `code-review` 或其他门禁

它不负责最终结论，也不负责重新规划。  
如果实施过程中发现范围变了、风险变了、计划失效了，它应该停下来，回到 `dev-planner`。

### `bug-fixer`

它负责“证据驱动修 bug”。

典型触发场景：

- 用户明确报告了 bug
- 已经有失败日志、复现步骤或失败测试
- `dev-builder` 的验证失败了
- `code-review` 给出了明确 findings，需要修

它不是“看到不顺眼就顺手改改”的 skill。  
它要求先有失败证据，再做根因判断，再做最小修复，最后重新复验。

一句人话：  
`bug-fixer` 解决的是“这次为什么坏、怎么最小修回去”，不是“顺便把整个模块重做一遍”。

### `code-review`

它负责“正式审查这次改动能不能过”。

典型触发场景：

- 一轮实施已经完成
- 已经补了验证证据
- 需要给出正式结论
- bug 修复后需要重新复审
- 发布前需要确认能不能进入交付

它的固定输出是：

1. `Findings`
2. `Risks not fully verified`
3. `Evidence checked`
4. `Final recommendation`

最后只允许 3 种结论：

- `Accept`
- `Hold`
- `Reject`

它不直接改代码。  
如果有实现问题，就把任务打回 `dev-builder` 或 `bug-fixer`。

### `release-builder`

它负责“发布或交付前的最后一层收口”。

典型触发场景：

- 需要正式发布
- 需要交付构建产物
- 需要部署或配置切换
- 需要做面向下游的版本更新

它关注的不是“功能能不能跑”，而是“这次能不能安全交出去”。

它会重点检查：

- 有没有 review 结论
- 有没有验证证据
- 有没有 `Release Card`
- 有没有冒烟检查
- 有没有回滚路径
- 是否要沉淀到 `runtime/releases/`、`runtime/errors/`、`runtime/adr/`

一句人话：  
`release-builder` 是交付门禁，不是平台型发布控制台。

## `.codex/` 是什么

这里放项目级 Codex 配置。  
当前最核心的入口是 `.codex/config.toml`。

你可以把这一层理解为：

- 它不是业务逻辑
- 它不是正式运行记录
- 它是这个项目对 Codex 的本地配置入口

所以它和 `runtime/` 的职责不一样。  
`runtime/` 记录任务怎么推进，`.codex/` 负责项目怎么接入 Codex。

## `docs/` 说明层在做什么

`docs/` 只负责解释规则，不负责承接运行中的正式对象。  
如果你把这里理解成“说明书层”，就对了。

### `docs/workflows/`

这是主流程骨架层，讲的是“任务怎么流”。

当前有 3 个核心流程：

#### `intake.md`

这是任务接入入口。  
它负责先判断：

- 当前任务是什么类型
- 当前 `Size` 是 `Small / Medium / Large`
- 当前 `Risk` 是 `Low / Medium / High`
- 要不要计划
- 会不会命中发布、安全、AI 行为等门禁

这里最重要的概念是 `Size + Risk` 双轴。  
`V3` 不是只看改动大小，还会看风险标签。  
高风险任务会自动抬级，所以“小改动但高风险”也不能按普通小改走。

#### `build-verify-review.md`

这是实施主链。  
它负责把下面这些动作连起来：

`implement -> verify -> review -> fix -> verify -> review`

这份流程说明的重点不是代码怎么写，而是：

- 动手前先确认边界
- 改完后必须有验证
- review 不是走过场
- fix 之后不能直接宣布完成

#### `release-retro.md`

这是交付与沉淀流程。  
它负责解释：

- 什么情况下进入正式发布路径
- 什么情况下要写 `Release Card`
- 什么情况下要记录到 `runtime/errors/`
- 什么情况下要写 `ADR-Lite`
- 什么时候应该把经验反哺回规则或模板

### `docs/protocols/`

这里放共享 gate 协议。  
它和 skill 的私有模板不同，只有“跨 skill 共用、必须统一口径”的内容才放这里。

当前你应该这样理解它：

- `adr-lite-template.md`：关键设计决策怎么记
- `security-checklist.md`：安全相关要补什么检查
- `ai-eval-template.md`：AI 行为变更要补什么评测证据

一句人话：  
`workflows` 讲顺序，`protocols` 讲格式和门禁字段。

## `runtime/` 运行层在做什么

`runtime/` 是 `V3` 最容易理解错的一层。  
它不是说明书，而是“当前任务真实承接和留痕的位置”。

如果你只记一句话，就记这个：

- `docs/` 讲规则
- `runtime/` 记这次任务真的发生了什么

### `runtime/PLANS.md`

这是单活跃长任务的计划入口。  
不是每个任务都要写它，只有这些场景才需要：

- 中改及以上
- 高风险抬级
- 跨会话承接
- 多阶段推进
- 暂停后恢复

它关心的字段包括：

- 任务目标
- 当前状态
- 范围 / 非目标
- 验收标准
- 风险与阻塞
- 验证方式
- 下一步

### `runtime/code_review.md`

这是默认 review 协议。  
它不等于某次 review 记录，而是告诉你正式 review 应该按什么结构输出。

### `runtime/plans/`

这是多活跃长任务模式下的计划目录。

#### `runtime/plans/active/`

放当前还在推进的多个计划。  
只有当单一 `runtime/PLANS.md` 已经不够承接时，才升级到这里。

#### `runtime/plans/archive/`

放已经完成或关闭的历史计划。  
它的作用是留档，不再继续写下一步。

### `runtime/reviews/`

放正式 review 记录。  
它不是 review 规则，而是“某一次审查的结果归档”。

### `runtime/errors/`

放错误、返工和判断失误记录。  
如果某类错误有复用价值，就应该沉淀到这里，避免下次再踩一样的坑。

### `runtime/releases/`

放正式发布或交付记录。  
这里承接的是 `Release Card`、冒烟结果、回滚路径和交付结论。

### `runtime/adr/`

放关键结构决策。  
不是所有实现细节都要写 ADR，只有会影响后续多轮迭代的重要决策才进来。

## `scripts/` 在体系里的位置

这里是全局脚本入口层。  
它的职责是辅助验证、安全检查和评测入口，不是主要真相源。

你现在只需要先知道：

- 它服务于 workflow 和 gate
- 它不是替代 `docs/` 的说明层
- 它也不是替代 `runtime/` 的记录层

所以阅读顺序上，它排在流程和 skill 之后。

## `src/`、`tests/`、`workspaces/` 怎么理解

### `src/`

这里放正式实现代码。  
它是项目真正落功能的地方。

### `tests/`

这里放传统验证内容。  
它的作用是证明“程序行为没被改坏”。

### `workspaces/`

这里放临时施工、草稿和试验。  
最关键的一条是：它不是真相源。  
定稿后要回填到正式目录，不能一直把重要内容挂在这里不归档。

## `configs/`、`prompts/`、`evals/` 是什么

这三层主要服务 `AI Pack`，不是每个项目一上来都要重度启用。

### `configs/`

放项目级配置物料。  
其中 `configs/models/` 主要承接模型路由、用途说明和评测关注点。

### `prompts/`

放正式 Prompt 物料。  
如果 Prompt 会影响项目行为，它应该是正式对象，不应该只放在聊天记录里。

### `evals/`

放 AI 行为评测。  
当 Prompt、模型、RAG、tool routing 或安全策略变化时，就会进入这一层。

一句人话：  
这三层是 AI 项目变重之后要用的正式目录，但没命中时不强制铺满。

## 一条完整任务通常怎么走

下面用“从接任务到收口”讲一遍：

### 第一步：任务接入

先读根级入口，再进入 `docs/workflows/intake.md`。  
这一步先不急着改代码，先判断：

- 任务类型
- `Size`
- `Risk`
- 下一步动作

如果这一步没做清楚，后面很容易一边做一边变质。

### 第二步：决定要不要计划

如果是很小、很稳的任务，可能只需要轻量边界说明。  
如果是中改、大改、高风险、跨会话任务，就要进 `dev-planner`，必要时写进 `runtime/PLANS.md`。

### 第三步：正式实施

边界清楚后，进入 `dev-builder`。  
这一步的重点不是“快改完”，而是“按边界做，不顺手扩写”。

### 第四步：补验证证据

改动完成后，进入验证。  
验证不是一句“我测过了”，而是要能说明：

- 跑了什么
- 结果怎样
- 还有什么没验证

### 第五步：正式 review

进入 `code-review`，输出结构化结论。  
如果发现问题，就回到 `dev-builder` 或 `bug-fixer`，然后重新 `verify -> review`。

### 第六步：需要交付时再进入 release

如果只是普通修复，可能到 review 通过就够了。  
如果要发布、部署、交付，就进入 `release-builder`，补 `Release Card`、冒烟检查和回滚路径。

### 第七步：按需沉淀

任务结束后，按需把内容写到：

- `runtime/reviews/`
- `runtime/errors/`
- `runtime/releases/`
- `runtime/adr/`

不是每次都要全写，但命中条件时不能省。

## 4 条最常见路径

### 路径 A：小改低风险

`README / AGENTS / PROJECT_RULES -> intake -> dev-builder -> verify -> code-review`

这是最轻的主链。  
不一定写长计划，也不一定进 release。

### 路径 B：中改或跨会话任务

`intake -> dev-planner -> runtime/PLANS.md -> dev-builder -> verify -> code-review`

这是正式计划路径。  
重点是让任务能被下一窗口或下一位执行者接住。

### 路径 C：明确 bug 或回归

`intake -> 固定失败证据 -> bug-fixer -> 复验 -> code-review`

这里最关键的是先有证据，再修。

### 路径 D：发布、交付或 AI 行为变更

`intake -> plan/build/review -> protocols gate -> release-builder / eval -> runtime 沉淀`

这里会更重，因为已经不是“单纯把功能改完”。

## 初学者最容易混淆的 6 件事

### 1. `docs/` 和 `runtime/` 不是一回事

`docs/` 负责解释规则。  
`runtime/` 负责记录这次任务真实发生了什么。

### 2. skill 不是脚本目录

skill 是任务角色和方法入口。  
脚本只是它可能用到的辅助工具。

### 3. 不是每个任务都写长计划

`V3` 强调按条件抬级，不强调一上来就写很多文档。

### 4. review 不是“看一眼就过”

它必须有固定结构和正式结论。

### 5. bug 修复不是独立王国

修完后还要回到 `code-review`，不能自己宣布结束。

### 6. `AI Pack` 不是默认全开

只有项目真的涉及 Prompt、模型、RAG、eval 时，`prompts/`、`configs/models/`、`evals/` 才会变成高频目录。

## 你现在可以怎么用这份说明

如果你的目标是“先看懂 `V3` 再检查它是否完整”，建议这样用：

1. 先通读本文，建立整体地图
2. 再打开 `V3-检查清单.md`
3. 按清单顺序回头检查 `README.md`、`AGENTS.md`、`PROJECT_RULES.md`、`docs/`、`runtime/`
4. 最后才深入具体 skill 正文

这样会比一上来扎进 `SKILL.md` 或脚本里更容易看懂。
