# V3 检查清单

## 文件定位

这份清单是 `V3` 的人工检查辅助文件。  
它默认帮助你按层检查结构、流程、skill 触发和运行对象，不展开脚本正文，也不逐字核 skill 本体。

如果清单和正式规则冲突，以 `README.md`、`AGENTS.md`、`PROJECT_RULES.md`、`docs/`、`runtime/` 为准。

## 使用方法

推荐从上往下检查，不要跳着看。  
前一层没确认清楚，后一层通常也很难判断对不对。

## 1. 根级入口层

- [ ] `README.md` 已清楚说明 `V3` 是什么、不是什么、适用项目、结构总览和默认用法。
- [ ] `AGENTS.md` 已清楚说明执行底线、抬级规则、验证要求、收口要求和记录路由。
- [ ] `PROJECT_RULES.md` 已清楚说明项目地图、任务路由、联动检查和阅读顺序。
- [ ] 根级没有堆放运行中的正式记录对象，入口层仍然足够轻。
- [ ] 根级新增的辅助文档已明确标注“导读 / 检查用途”，没有冒充新的真相源。

## 2. 总分层边界

- [ ] 当前结构仍然符合“根级入口层 + `docs/` 说明层 + `runtime/` 运行层”。
- [ ] `docs/` 里只放解释型说明，没有混入运行中的正式记录。
- [ ] `runtime/` 里只放运行对象，没有把说明文档塞进来。
- [ ] `.agents/skills/` 仍然是 `skill package first`，单 skill 私有模板和私有脚本没有重新散回全局。
- [ ] 没有新增平行目录、平行真相源或重复副本。

## 3. `Core + Gates + Packs` 口径

- [ ] `Core` 被解释为默认最小正式闭环。
- [ ] `Gates` 被解释为命中条件后必须加上的正式门禁。
- [ ] `Packs` 被解释为按需启用的扩展层，而不是默认全开。
- [ ] `V3` 没有被写成平台型 runtime，也没有被写成默认多 agent 控制面。
- [ ] `Size + Risk` 双轴判断仍然是正式入口，不是可选说明。

## 4. `docs/workflows/` 主链

- [ ] `intake.md` 已说明任务类型、`Size + Risk`、抬级规则、计划判断和下一步动作。
- [ ] `build-verify-review.md` 已明确 `implement -> verify -> review -> fix -> verify -> review`。
- [ ] `build-verify-review.md` 已说明失败证据优先和验证层次。
- [ ] `release-retro.md` 已说明发布、交付、沉淀、复盘和规则升级顺序。
- [ ] 三条 workflow 之间的进入条件和出口产物能接得上，没有断链。

## 5. `docs/protocols/` 共享门禁

- [ ] `docs/protocols/README.md` 已明确这里只放共享 gate，而不是 skill 私有模板。
- [ ] `adr-lite-template.md` 的定位仍然是关键决策记录协议。
- [ ] `security-checklist.md` 的定位仍然是安全检查门禁。
- [ ] `ai-eval-template.md` 的定位仍然是 AI 行为评测门禁。
- [ ] skill 私有模板没有重新复制回 `docs/protocols/`。

## 6. 5 个核心 skill 的触发与职责

- [ ] `dev-planner` 仍然负责“是否需要计划、计划写到什么程度、是否沉淀到 `runtime/`”。
- [ ] `dev-planner` 仍然保留 `极小 / 小 / 中 / 大` 四档路由。
- [ ] `dev-builder` 仍然只负责实施、补验证证据和把结果送去后续门禁，不代替 planner 和最终收口。
- [ ] `bug-fixer` 仍然坚持“先有失败证据，再做根因修复”，不是顺手重构 skill。
- [ ] `code-review` 仍然只输出结构化审查结论，不直接偷偷改代码。
- [ ] `code-review` 的固定输出仍然是 `Findings / Risks not fully verified / Evidence checked / Final recommendation`。
- [ ] `release-builder` 仍然是交付 / 发布门禁，不是重型多平台发布控制面。
- [ ] `release-builder` 仍然要求 `Release Card`、冒烟检查和回滚路径。

## 7. `runtime/` 运行对象层

- [ ] `runtime/README.md` 已明确这里是运行层，不是说明层。
- [ ] `runtime/PLANS.md` 已明确用于单活跃长任务，不是所有任务默认必写。
- [ ] `runtime/code_review.md` 已明确是 review 协议，不是某次 review 记录本身。
- [ ] `runtime/plans/` 已明确只在多活跃长任务模式下启用。
- [ ] `runtime/plans/active/` 只放进行中的多计划对象。
- [ ] `runtime/plans/archive/` 只放已完成或已关闭的历史计划。
- [ ] `runtime/reviews/` 只放正式审查记录。
- [ ] `runtime/errors/` 只放错误、返工和判断失误记录。
- [ ] `runtime/releases/` 只放正式发布或交付记录。
- [ ] `runtime/adr/` 只放关键结构决策。

## 8. 常规工程目录

- [ ] `src/` 仍然被解释为正式实现代码目录。
- [ ] `tests/` 仍然被解释为传统验证目录。
- [ ] `workspaces/` 仍然被明确标成临时施工区，不是真相源。
- [ ] `scripts/` 仍然被解释为全局验证 / 安全 / eval 入口层，而不是说明层或记录层。
- [ ] `.codex/config.toml` 仍然被定位为项目级 Codex 配置入口。

## 9. `AI Pack` 相关目录

- [ ] `configs/` 已明确只放项目级配置物料，不放真实密钥。
- [ ] `configs/models/` 已明确只在模型路由或模型行为变化时重点使用。
- [ ] `prompts/` 已明确是正式 Prompt 物料目录，不应长期停留在临时区。
- [ ] `evals/` 已明确用于 AI 行为评测，而不是普通代码验证。
- [ ] 这些目录仍然保持“按需启用”，没有被写成默认必须铺满。

## 10. 任务主流程检查

- [ ] 小改低风险任务可以走轻链路，不被机械要求写长计划。
- [ ] 中改、大改、高风险、跨会话任务会被抬级到正式计划路径。
- [ ] bug、回归、失败验证场景会先固定失败证据，再进入修复。
- [ ] 修复完成后会重新进入 `verify -> review`，不会跳过复审。
- [ ] 发布、交付、配置切换、AI 行为变更会显式进入对应 `Gate`。

## 11. 术语与路由一致性

- [ ] `Core`、`Gates`、`Packs` 三个术语在根级、workflow、skill、设计文档里保持一致。
- [ ] `Size + Risk` 的判断逻辑在根级和 workflow 中保持一致。
- [ ] `Verification Evidence` 的结构说明前后一致。
- [ ] `Accept / Hold / Reject` 的定义前后一致。
- [ ] `Release Card`、`ADR-Lite`、`Security Lite`、`AI Eval Gate` 的命名和路径没有漂移。

## 12. 真相源与记录规则

- [ ] 当前用户要求仍然放在最高优先级。
- [ ] 根级入口、`docs/`、`runtime/` 的真相源顺序没有被新文档打乱。
- [ ] 正式记录都落在 `runtime/` 或仓库级 `docs/errors/`、`docs/logs/` 规则要求的位置。
- [ ] 明确错误、返工或判断失误已按规则记录到 `docs/errors/`。
- [ ] 当日有实际任务时，`docs/logs/` 已追加日志。
