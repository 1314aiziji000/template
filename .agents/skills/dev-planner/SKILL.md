name: dev-planner
description: 当任务需要先规划再实施时使用。先按 `V3` 的 `intake` 路由判断任务类型与 `Size + Risk`，再进入四档筛选：`极小 / 小 / 中 / 大`。`极小` 与 `小` 使用仓库内轻计划；`中` 与 `大` 优先使用 Codex 官方计划能力。只有命中跨会话、多阶段、暂停恢复或正式交接等条件时，才把计划沉淀到 `runtime/PLANS.md`；必要时再升级到 `runtime/plans/active/*.md`。

[任务]
    **极小处理模式**：当前任务只是一次性单点修改、边界稳定、无需正式承接时，不进入官方重计划，也不默认写 `runtime/PLANS.md`。只输出边界、非目标、最小验证和下一步，直接交给实施阶段。

    **小型轻计划模式**：当前任务仍属 `Small + Low Risk`，但值得先想清楚再动手时，使用本 Skill 的轻计划方式。重点说清任务目标、范围、非目标、最小验证和下一步；默认不沉淀到 `runtime/PLANS.md`，除非用户明确要求或后续命中沉淀条件。

    **中大型官方优先模式**：当前任务进入 `Medium`、`Large` 或高风险抬级路径时，优先使用 Codex 官方计划能力。官方计划负责把当前实现路线想清楚；本 Skill 负责判断是否需要把这份计划进一步沉淀到仓库的正式计划对象。

    **计划沉淀模式**：当官方计划或轻计划需要跨会话、多阶段、暂停恢复或正式交接时，把当前已明确的计划整理为 `runtime/PLANS.md`。如单一计划不再够用，再升级为：
    - `runtime/PLANS.md`：总览与索引
    - `runtime/plans/active/*.md`：活跃计划
    - `runtime/plans/archive/*.md`：已收口计划

    **更新计划模式**：当已有计划对象的范围、风险、阶段状态、验证路线或下一步发生变化时，更新现有 `runtime/PLANS.md` 或 `runtime/plans/active/*.md`。已完成阶段只更新状态和影响，不重写成一份新计划。

[依赖检测]
    Skill 启动时第一步自动执行：

    必需：
    - 当前任务要求
    - `README.md`、`AGENTS.md`、`PROJECT_RULES.md`
    - `docs/workflows/intake.md`
    - `.agents/skills/dev-planner/templates/plans-template.md`

    按需必读：
    - 已存在的 `runtime/PLANS.md` → 有则必须读取，判断当前是不是新建、更新还是升级
    - 已存在的 `runtime/plans/active/`、`runtime/plans/archive/` → 有则必须读取，避免平行计划
    - 当前任务相关的正式文档、现有实现、验证结果、`runtime/reviews/`、`runtime/errors/`、`runtime/releases/`、`runtime/adr/`

    当前会话可复用：
    - 当前会话里已经形成的官方计划内容 → 可作为本 Skill 的输入
    - 但它不是仓库真相源，不自动等于 `runtime/PLANS.md`

    可选（必要时核验）：
    - 当前代码与测试结构 → 用于识别关键文件、模块边界和现有约束
    - 官方文档、正式资料或最新依赖说明 → 只在技术方案不确定、外部依赖可能过期、兼容性存在疑问时核验

    不作为默认前置：
    - `Product-Spec.md`
    - `Design-Brief.md`
    - 设计稿 MCP

[第一性原则]
    **先 intake，再筛档**：`dev-planner` 不是脱离上下文的独立规划器。即使用户直接调用，也必须先按 `docs/workflows/intake.md` 判断任务类型、`Size`、`Risk` 和是否命中正式计划条件，再决定走 `极小 / 小 / 中 / 大` 哪一档。

    **官方优先，但不替代仓库承接**：`中`、`大` 任务优先用 Codex 官方计划能力；本 Skill 不和官方计划引擎抢角色。它负责把“这次该不该沉淀成仓库计划对象”判断清楚。

    **官方计划不等于仓库计划文档**：当前会话里的官方计划通常可读，但它不是稳定的仓库真相源。只要任务要跨会话、多阶段、暂停恢复或正式交接，就必须沉淀到 `runtime/PLANS.md` 或 `runtime/plans/active/*.md`。

    **计划是活文档**：一旦计划已沉淀进 `runtime/`，它的目标不是写得长，而是让后来者只看计划也能继续推进。阶段推进、风险变化、范围变化、验证结果变化后，必须更新。

    **轻量规格补足**：`V3` 不单独引入重型 `SPEC.md`。计划里必须用 `非目标`、`验收标准`、`边缘场景` 补足目标定义和回归依据，避免“只会拆任务，不会定义边界”。

    **依赖正序**：先解决会影响全局路径判断的前置约束，再拆后续实施阶段。被依赖的基础项先做，高风险和高不确定项尽量前置。

    **必要时核验**：不靠过期记忆规划外部依赖。只有在技术方案不确定、外部依赖可能过期或集成兼容性存在疑问时，才查官方资料或正式来源，不默认铺开重型技术选型分析。

    **边界先于动作**：无论走哪一档，计划都必须先说清范围、非目标、验证方式和下一步。没有边界的阶段拆解只是换一种写法的待办清单。

    **无占位符原则**：计划正文不允许 `TBD`、`TODO`、`待补充`、`待确定`、`后面再说`、`类似上一阶段` 这类空话。每个阶段都要具体到后来者能直接继续做。

[输出风格]
    **语态**：
    - 像给下一位接手的 Codex 或工程师写交接说明
    - 简洁、可续跑、能落到文件和验证动作

    **原则**：
    - x 不把计划写成重型产品文档
    - x 不把实现细节、函数签名、样式方案塞进计划
    - x 不在 `极小`、`小` 场景下机械生成长计划
    - x 不把官方计划内容原样复制成聊天纪要式文档
    - x 不在已有计划仍有效时另起平行计划
    - ✓ 先给出筛档结论，再决定计划落点
    - ✓ 官方计划优先场景要明确说明是否需要沉淀到 `runtime/`
    - ✓ 一旦沉淀到 `runtime/PLANS.md`，字段必须和固定协议一致
    - ✓ 输出必须能直接接到 `dev-builder` 或对应 `Gate`

    **典型表达**：
    - "当前任务判定为 `极小修改`，不进入正式计划文档，只保留边界、最小验证和下一步。"
    - "当前任务判定为 `小型修改`，走仓库轻计划，不默认沉淀到 `runtime/PLANS.md`。"
    - "当前任务判定为 `中型修改`，优先使用官方计划能力；如果本次无法在单窗口内完成，再沉淀到 `runtime/PLANS.md`。"
    - "当前任务判定为 `大型修改`，官方计划优先，且已命中多阶段推进，必须沉淀到 `runtime/PLANS.md`。"
    - "当前已出现两个并行长任务，单一 `runtime/PLANS.md` 不再够用，升级为总览 + `runtime/plans/active/*.md`。"

    └── templates/

    - `runtime/plans/active/*.md`

[分析维度清单]
    分析当前任务时，需要覆盖以下维度：

    **必须分析**（没有这些，就不算正式进入计划判断）：

    - 任务类型：新功能、bug、重构/迁移、发布/交付、AI 行为变更中的哪一类
    - `Size + Risk`：至少得出当前 `Size`、`Risk` 和是否需要抬级
    - 当前筛档：这次属于 `极小 / 小 / 中 / 大` 中哪一档
    - 是否需要正式计划沉淀：当前任务只需会话内计划，还是必须沉淀到 `runtime/`
    - 任务目标：这次到底要完成什么，收口后什么算达成
    - 当前状态：进行中、待开始、暂停中、待验证、待 review、待 release 中的哪一种
    - 范围与非目标：当前明确要做什么，不做什么
    - 验收标准与边缘场景：至少能说明怎么判断完成、哪些边界需要回归
    - 关键文件 / 模块：计划会影响哪些正式实现、配置、脚本或运行对象
    - 验证方式：最小验证、回归验证、失败证据优先是否触发
    - 风险与阻塞：当前已知风险、外部依赖、未知点和等待项
    - 下一步：计划生成后马上该做什么，由谁接续

    **需要沉淀时必须分析**：
    - 实施阶段拆解：阶段顺序要符合依赖关系，每一阶段都能独立接续
    - 当前会话里的官方计划已经明确到什么程度
    - 是否存在沉淀触发器：跨会话、多阶段、暂停恢复、交接、范围扩大、风险变化、验证结果改变路线、用户要求正式承接
    - 是否需要拆成多计划：单一 `runtime/PLANS.md` 还能否清楚承接

    **不需要分析**（交给其他 skill 或后续阶段）：
    - 具体代码实现细节
    - 具体测试用例设计
    - 具体发布执行步骤
    - 具体 prompt、模型、RAG 配置内容

[分析策略]
    **四档筛选法**
    先对照 `docs/workflows/intake.md` 做正式路由，再落到 4 个执行档位：
    1. `极小`：单点修正、边界清楚、通常 `1` 个文件、可当次做完、无需正式承接
    2. `小`：仍属 `Small + Low Risk`，但值得先给一份轻计划再动手
    3. `中`：`Medium`，或 `Small` 因高风险抬级后按中型执行
    4. `大`：`Large`，或 `Medium` 因高风险抬级后按大型执行

    说明：
    - `极小` 是本 Skill 的路由层特例，不替代 `V3` 正式的 `Small / Medium / Large` 体系
    - `中`、`大` 是“官方计划优先”的信号，不等于一定立即写 `runtime/PLANS.md`

    **轻重分流法**
    - `极小`：直接给边界、非目标、最小验证、下一步
    - `小`：给轻计划，不默认沉淀
    - `中`、`大`：优先使用官方计划能力，再判断是否沉淀到仓库计划文档

    **沉淀触发法**
    出现下面任一情况时，必须把当前计划沉淀到 `runtime/`：
    - 需要跨会话承接
    - 预计会暂停恢复
    - 需要多阶段推进
    - 需要正式交接给下一位执行者
    - 验证结果会改变下一步路线
    - 任务范围明显扩大
    - 风险标签发生变化
    - 用户明确要求形成正式计划文档

    **会话优先读取官方计划法**
    如果当前会话里已经有官方计划内容：
    - 优先复用这份计划作为分析输入
    - 再判断是否需要沉淀
    - 沉淀时只保留仓库承接真正需要的结构化内容，不复制整段对话

    **边界围栏法**
    从目标反推范围，再用 `非目标` 把不会做的内容写清。凡是容易“顺手扩写”的点，都优先写进 `非目标`，避免后续实施跑偏。

    **阶段可接续法**
    只有在需要沉淀到 `runtime/` 时，才拆完整实施阶段。每个阶段都必须回答三件事：
    1. 这一阶段完成什么
    2. 这一阶段主要动哪些文件 / 模块
    3. 这一阶段完成后怎么验证
    任何一阶段如果不能独立接续，就要继续拆。

    **更新优先法**
    如果已有 `runtime/PLANS.md`，默认先更新，不新建平行计划。只有当出现多个活跃长任务，才升级到 `runtime/plans/active/*.md`。

    **验证倒推法**
    先想清楚收口时要拿什么证据，再反推阶段拆解。需要失败证据优先的场景，计划里必须先写“先拿证据，再进入实现”。

    **必要时核验法**
    只有下面这些情况才查外部正式资料：
    - 技术方案存在多个合理选项
    - 外部依赖版本、接口、规则明显可能过期
    - 兼容性问题会直接影响计划顺序
    核验后只保留影响计划的结论，不把外部资料整理成重型技术报告。

[信息充足度判断]
    当以下条件满足时，可以完成当前档位对应的计划输出：

    **`极小` 必须满足**：
    - 当前边界已明确
    - 非目标已明确
    - 最小验证已明确
    - 下一步动作明确

    **`小` 必须满足**：
    - `Size + Risk` 已判断
    - 当前目标、范围、非目标已明确
    - 最小验证或回归验证方式已明确
    - 下一步动作明确

    **`中`、`大` 必须满足**：
    - 已判断当前任务应优先走官方计划能力
    - 当前会话里的实现路线已足够清楚
    - 已判断是否命中沉淀条件

    **需要沉淀到 `runtime/` 时必须满足**：
    - 任务目标、当前状态、范围、非目标已明确
    - 验收标准、边缘场景、关键文件 / 模块已明确
    - 阶段拆解和验证方式已足够让后来者继续做
    - 下一步动作明确

    如果上述条件未达成：
    - 继续读仓库真相源和现有实现
    - 必要时向用户确认高影响歧义
    - 不生成半成品计划文档

[工作流程（新建计划模式）]
    [加载阶段]
        目的：建立 `V3 / Codex` 上下文，不跳过根级和 workflow 路由

        第一步：依赖检测
            执行 [依赖检测]

        第二步：加载仓库级真相源
            读取：
            - `README.md`
            - `AGENTS.md`
            - `PROJECT_RULES.md`
            - `docs/workflows/intake.md`
            - `.agents/skills/dev-planner/templates/plans-template.md`

        第三步：加载当前运行状态
            检查：
            - `runtime/PLANS.md`
            - `runtime/plans/active/`
            - `runtime/plans/archive/`
            - 与当前任务直接相关的 `runtime/reviews/`、`runtime/errors/`、`runtime/releases/`、`runtime/adr/`

        第四步：加载当前任务材料
            读取当前任务要求、相关实现、验证结果和正式文档
            识别这次任务真实会影响的模块、配置和运行对象

        第五步：读取当前会话中的官方计划（如有）
            如果当前会话里已经进入官方计划流程并已有可用计划内容：
            - 读取其已明确的实现路线
            - 作为后续筛档和沉淀判断输入

    [路由阶段]
        目的：先做 `intake` 和四档筛选，再决定计划落点

        第一步：判断任务类型
            将当前任务归入：
            - 新功能或需求变更
            - bug / 回归 / 异常
            - 重构 / 结构调整 / 迁移
            - 发布 / 交付 / 配置切换
            - AI 行为变更

        第二步：判断 `Size + Risk`
            对照 `docs/workflows/intake.md`：
            - 判断当前 `Size`
            - 判断当前 `Risk`
            - 如为高风险，先抬级一个执行等级

        第三步：执行四档筛选
            - 命中 `极小` 信号 → 走极小处理模式
            - `Small + Low Risk` 但值得先想清楚 → 走小型轻计划模式
            - `Medium` 或抬级后按中型执行 → 走中型官方优先模式
            - `Large` 或抬级后按大型执行 → 走大型官方优先模式

        第四步：判断是否命中沉淀条件
            对照 [沉淀触发法]：
            - 未命中 → 不写 `runtime/PLANS.md`
            - 已命中 → 进入计划沉淀模式

    [分析阶段]
        目的：把“该走哪一档”推进到“当前计划该输出什么”

        第一步：明确边界
            写清：
            - 任务目标
            - 当前状态
            - 范围
            - 非目标

        第二步：明确收口依据
            写清：
            - 验收标准
            - 边缘场景
            - 验证方式

        第三步：识别关键实现边界
            标出：
            - 关键文件 / 模块
            - 当前风险与阻塞
            - 是否需要失败证据优先
            - 是否命中其他 `Gate`

        第四步：按档位决定计划深度
            - `极小`：不拆长阶段，只给边界、最小验证和下一步
            - `小`：给轻计划，不默认写 `runtime/PLANS.md`
            - `中`、`大`：优先复用官方计划内容；如命中沉淀条件，再拆完整实施阶段

        第五步：必要时核验外部信息
            仅在 [必要时核验法] 命中的情况下，查官方资料或正式来源
            只保留会影响计划顺序、风险判断或验证方式的结论

    [输出阶段]
        目的：按当前档位输出可直接接续的计划结果

        第一步：输出 `极小` / `小` 结果
            - `极小`：输出边界、非目标、最小验证、下一步
            - `小`：输出轻计划，默认不写 `runtime/PLANS.md`

        第二步：输出 `中` / `大` 结果
            - 明确说明当前任务应优先使用官方计划能力
            - 若当前会话里已存在官方计划，则复用其结果
            - 若未命中沉淀条件，则保持为会话内计划，不写 `runtime/PLANS.md`

        第三步：需要沉淀时对齐固定字段
            对照 `.agents/skills/dev-planner/templates/plans-template.md` 和 `runtime/PLANS.md`
            确保计划至少包含：
            - 任务目标
            - 当前状态
            - 范围
            - 非目标
            - 验收标准
            - 边缘场景
            - 关键文件 / 模块
            - 风险与阻塞
            - 实施阶段拆解
            - 验证方式
            - 下一步

        第四步：自检
            检查：
            - 没有占位符和空话
            - 没有漏掉筛档结论
            - 一旦写入 `runtime/`，字段与固定协议一致
            - 如果选择不沉淀，理由明确
            - 下一步可直接交给 `dev-builder` 或对应 `Gate`

        第五步：输出计划对象
            - 单任务：写入或更新 `runtime/PLANS.md`
            - 多任务：升级为 `runtime/PLANS.md` 总览 + `runtime/plans/active/*.md`

        第六步：指向下一步
            明确说明：
            - 当前下一步动作
            - 是否交给 `dev-builder`
            - 是否需同步进入 review / release / AI eval 等门禁

[工作流程（更新计划模式）]
    触发条件：
    - 已沉淀的计划对象存在
    - 阶段推进了
    - 风险判断变了
    - 实施范围变了
    - 验证结果改变了下一步
    - 发现原计划漏了关键约束

    [更新阶段]
        第一步：加载现有计划
            读取现有 `runtime/PLANS.md`
            如已升级为多计划模式，则同时读取：
            - `runtime/plans/active/*.md`
            - `runtime/plans/archive/*.md`

        第二步：比对当前状态
            对照最新任务状态、验证结果、风险变化和现有实现
            标出哪些字段需要更新，哪些阶段已完成，哪些下一步已失效

        第三步：只更新受影响部分
            重点更新：
            - 当前状态
            - 范围 / 非目标
            - 风险与阻塞
            - 实施阶段拆解
            - 验证方式
            - 下一步

            已完成阶段：
            - 保留完成事实
            - 只补状态或影响说明
            - 不重写成新计划

        第四步：重新校验路由
            检查这次更新后是否需要同步进入：
            - `runtime/reviews/`
            - `runtime/releases/`
            - `AI Eval Gate`
            - 多计划模式

[工作流程（多计划升级模式）]
    触发条件：
    - 出现多个活跃长任务
    - 单一 `runtime/PLANS.md` 已无法同时清楚承接多个任务
    - 当前计划之间存在不同状态、不同验证路径或不同下一步

    [升级阶段]
        第一步：确认升级必要性
            只有在单计划确实不够用时，才升级
            不为了“结构更完整”而机械创建多计划目录

        第二步：保留总览
            `runtime/PLANS.md` 只保留：
            - 当前活跃计划索引
            - 每个计划的状态
            - 总体下一步或注意事项

        第三步：拆出活跃计划
            把每个活跃长任务写入 `runtime/plans/active/*.md`
            每个活跃计划仍需保持 `PLANS` 固定字段完整

        第四步：归档已完成计划
            已完成或已关闭的计划移入 `runtime/plans/archive/`
            避免活跃目录混入历史对象
