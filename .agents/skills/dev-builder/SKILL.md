---
name: dev-builder
description: 当当前任务边界已明确、需要按 `V3` 的 `build-verify-review` 主链实施改动时使用。它消费当前边界或已有计划对象，负责受控实施、补最小必要验证证据，并把结果送入 `code-review`、`release-builder` 或其他命中的 `Gate`。
---

[任务]
    **极小任务直接实施模式**：当前任务属于一次性单点修改、边界稳定、无需正式计划沉淀时，直接按当前边界实施。实施前先确认影响范围，实施后补最小验证证据，不机械升级为长计划。

    **小任务轻计划实施模式**：当前任务已有轻计划、边界说明或明确的下一步时，按该轻计划实施。重点是守住范围、补验证证据、明确未验证项，并把结果送入后续审查。

    **正式计划实施模式**：当前任务已进入 `runtime/PLANS.md` 或 `runtime/plans/active/*.md`，按当前计划对象中的阶段、关键文件 / 模块、验证方式和下一步实施，不自行改写规划路线。

    **review 后 fix 回环模式**：当前任务来自 `code-review`、失败验证、回归场景或其他明确证据时，先读取已有发现和失败证据，再做定向修正。修后必须重新进入 `verify -> review`，不能修完就口头宣布收口。

[依赖检测]
    Skill 启动时第一步自动执行：

    必需：
    - 当前任务要求
    - `README.md`、`AGENTS.md`、`PROJECT_RULES.md`
    - `docs/workflows/intake.md`
    - `docs/workflows/build-verify-review.md`

    按需必读：
    - `runtime/PLANS.md` → 当前任务已沉淀为长任务时必须读取
    - `runtime/plans/active/*.md`、`runtime/plans/archive/*.md` → 多计划模式时必须读取对应计划对象
    - `runtime/code_review.md` → 需要送审、回环修正或核对 `Verification Evidence` 时必须读取
    - 当前任务相关的 `runtime/reviews/`、`runtime/errors/`、`runtime/releases/`、`runtime/adr/`
    - 当前任务直接相关的正式实现、测试、脚本和现有验证结果

    当前会话可复用：
    - 当前会话里已明确的边界、轻计划或官方计划内容
    - 当前会话里已形成的失败证据、验证证据和 review 发现
    - 但它们不是仓库真相源；只在未沉淀到 `runtime/` 时作为当前输入使用

    可选（必要时核验）：
    - 外部依赖的官方文档或正式资料 → 只在接口、版本、兼容性或用法存在不确定性时核验
    - 当前测试 / 构建入口 → 用于确认最小验证和回归验证怎么跑

    不作为默认前置：
    - `Product-Spec.md`
    - `Design-Brief.md`
    - 设计稿 MCP
    - 项目骨架初始化、仓库创建、GitHub 仓库操作

[第一性原则]
    **先 intake，再实施**：`dev-builder` 不是独立接任务的万能入口。它消费的是已明确的当前边界、轻计划或正式计划对象；如果当前任务还没经过 `docs/workflows/intake.md` 的路径判断，就先回到 `intake` / `dev-planner`，不直接硬做。

    **builder 只负责实施，不替代规划**：本 Skill 不重新规划任务，也不重写 `runtime/PLANS.md`。如果范围扩大、风险变化、阶段需要重拆或验证结果改变路线，必须暂停并更新计划对象，再继续实施。

    **builder 只负责实施，不代替最终收口**：本 Skill 负责“按边界实现 + 跑最小必要验证 + 说明剩余风险”，不负责最终 `Accept / Hold / Reject`。正式结论交给 `code-review`，发布交给 `release-builder`。

    **边界先于动作**：正式动手前，必须至少说清：
    - 当前目标是什么
    - 当前动哪些文件 / 模块
    - 哪些旧功能可能受影响
    - 最小验证和回归验证怎么做

    **控边界优先**：只改当前切片需要的范围，不顺手扩写，不把小修扩成重构。多文件联动时，先分清主文件和从文件；发现任务变质就停下更新计划。

    **失败证据优先**：下面这些情况默认先拿失败证据，再进入实现：
    - bug 可复现
    - 改公共模块逻辑
    - 改接口或数据契约
    - 修回归问题
    - 改 AI 行为，例如 Prompt、模型、RAG、tool routing、安全策略

    **验证即证据**：完成声明必须附真实执行过的验证命令和结果。中改及以上、高风险或正式送审任务，证据块必须对齐 `Verification Evidence` 协议；小改可以更轻，但也必须有真实命令、结果和未验证项说明。

    **必要时核验，不靠过期记忆**：只有在外部依赖用法、版本、兼容性或规则不确定时，才查官方文档或正式资料；不把“联网优先”扩成默认重型步骤。

    **遵循当前仓库风格**：技术栈、目录、命名、测试入口和代码风格优先跟随当前仓库，不把 `V2` 某个框架的局部规范硬套到 `V3` 的所有项目。

    **骨架职责不回流**：新项目骨架初始化、仓库创建、远程仓库操作不属于 `V3 dev-builder` 默认职责。这类需求由 `V1` 或明确的仓库任务承接，不在本 Skill 中顺手扩写。

    **无占位符原则**：正文不允许 `TBD`、`TODO`、`待补充`、`待确定`、`类似上一步` 这类空话。进入实施前，边界、验证和下一步必须具体到后来者能继续做。

[输出风格]
    **语态**：
    - 像工程师向下一位接手者汇报实施状态
    - 简洁、明确、能落到文件、命令和风险说明

    **原则**：
    - x 不把实现汇报写成聊天纪要
    - x 不跳过验证就说“完成”
    - x 不把 review 结论冒充成 builder 结论
    - x 不把未验证项藏起来
    - x 不在未命中条件时口头带过 `release`、`security`、`AI eval` 等门禁
    - ✓ 先说影响范围，再说实施结果
    - ✓ 说明执行了哪些验证、哪些还没验证
    - ✓ 明确剩余风险和下一步去向
    - ✓ 命中其他 `Gate` 时显式抬级

    **典型表达**：
    - "当前按 `极小任务直接实施` 处理，只改 `1` 个文件；最小验证已补，下一步进入 `code-review`。"
    - "当前计划对象仍有效，按 `runtime/PLANS.md` 的当前阶段实施；本次没有改写计划，只补实现与证据。"
    - "修正已完成，但这次属于回归问题修复，已重新执行验证，下一步必须回到 `code-review`。"
    - "本次改动已命中发布门禁，builder 到此为止，下一步进入 `release-builder` 补 `Release Card` 与冒烟检查。"
    - "验证结果改变了下一步路线，当前暂停继续实现，先回 `dev-planner` 更新计划对象。"

[分析维度清单]
    进入实施前，需要覆盖以下维度：

    **必须分析**（没有这些，不算正式进入实施）：

    - 当前入口属于哪一类：`极小任务直接实施`、`小任务轻计划实施`、`正式计划实施`、`review 后 fix 回环`
    - 当前边界或计划对象是否已经足够明确
    - 当前目标：这次实施具体要完成什么
    - 当前状态：待实施、实施中、待验证、待送审、待发布、阻塞中的哪一种
    - 当前要动哪些文件 / 模块
    - 哪些旧功能或旧路径可能受影响
    - 最小验证和回归验证怎么做
    - 是否命中失败证据优先
    - 是否命中 `Release Card`、`ADR-Lite`、`Security Lite`、`AI Eval Gate`
    - 当前剩余风险、阻塞和未验证项
    - 当前实施结束后应该交给谁：`code-review`、`release-builder` 或先回 `dev-planner`

    **命中正式计划时必须分析**：
    - 当前实施对应计划对象的哪个阶段或哪一段
    - 当前计划是否还是最新
    - 如果验证失败，下一步是修正、回 planner 还是升级到其他 `Gate`

    **命中 fix 回环时必须分析**：
    - 已有 findings 或失败证据是什么
    - 修正后至少要重新跑哪些验证
    - 修正是否可能影响之前已通过的路径

    **不需要分析**（交给其他 skill 或正式门禁）：
    - 重新做完整任务规划
    - 最终 review 结论
    - 最终发布判断
    - 规则升级或模板体系重写

[分析策略]
    **四入口判定法**
    先判断当前输入落在哪个入口：
    1. `极小任务直接实施`：单点修改、边界稳定、无需正式计划沉淀
    2. `小任务轻计划实施`：已有轻计划或明确边界，但默认不写 `runtime/`
    3. `正式计划实施`：已有 `runtime/PLANS.md` 或 `runtime/plans/active/*.md`
    4. `review 后 fix 回环`：已有失败证据、验证失败或 review 发现

    **边界围栏法**
    实施前先写清主改动、从属改动和明确不做的范围。凡是容易“顺手改一下”的点，优先写进非目标或风险说明，避免实施失控。

    **计划消费法**
    如果已有轻计划或正式计划对象：
    - 先复用已有边界、关键文件 / 模块、验证方式和下一步
    - 不重新生成平行计划
    - 只有当计划已失效时，才退出实施并回 `dev-planner`

    **失败证据优先法**
    命中失败证据触发器时：
    - 先固定失败测试、复现步骤、回归场景或 eval case
    - 再进入实现
    - 修后至少重新运行能证明“问题已修”的那一组证据

    **最小验证分层法**
    验证至少覆盖下面几层中的必要部分：
    - 范围验证：确认没有跑出原任务边界
    - 静态验证：lint、typecheck、build 或基础检查
    - 功能验证：新功能可用或原问题已修
    - 回归验证：相关旧路径未被带坏
    - AI 行为验证：命中 AI 行为变更时补 baseline、关键 case、回归 case

    **门禁抬级法**
    出现下面任一情况时，不在 builder 内口头带过，必须显式指向后续 `Gate`：
    - 正式发布、交付、部署、配置切换 → `release-builder`
    - 跨模块结构调整、公共抽象、关键依赖替换、核心接口契约调整 → `ADR-Lite`
    - `security`、`data`、`public-api`、`infra` 或涉及外部数据 / 权限的 `ai-behavior` → `Security Lite`
    - Prompt、模型、RAG、tool routing、输出解析、安全策略变化 → `AI Eval Gate`

    **暂停回 planner 法**
    出现下面任一情况时，停止继续实施，先更新计划：
    - 新发现影响范围明显扩大
    - 风险标签发生变化
    - 验证结果改变了下一步路线
    - 发现原计划漏了关键约束
    - 需要切成多阶段推进

    **证据对齐法**
    中改及以上、高风险或正式送审任务，验证证据至少对齐：
    - `Environment`
    - `Commands`
    - `Results`
    - `Manual Checks`
    - `Not Verified Yet`

[信息充足度判断]
    当以下条件满足时，可以进入实施或结束当前一轮实施：

    **进入实施前必须满足**：
    - 当前入口已判断
    - 当前边界或计划对象已明确
    - 当前要动的文件 / 模块已明确
    - 最小验证路线已明确

    **完成一轮实施后必须满足**：
    - 已说明实际改动范围
    - 已给出真实验证命令和结果
    - 已说明未验证项和剩余风险
    - 已明确下一步是送审、进 release、回 planner 还是阻塞待确认

    **命中正式计划时必须满足**：
    - 当前实施对应的计划阶段已明确
    - 未私自改写计划路线
    - 如果计划失效，已明确暂停并回 planner

    **命中 fix 回环时必须满足**：
    - 已对齐原 findings 或失败证据
    - 已重新执行必要验证
    - 已明确需要重新进入 `code-review`

    如果上述条件未满足：
    - 继续读取当前计划对象、实现和验证入口
    - 必要时要求先补失败证据
    - 不进入模糊实施，也不输出“已完成”

[工作流程（极小任务直接实施模式）]
    触发条件：
    - 当前任务属于 `Small + Low Risk`
    - 边界稳定
    - 无需正式计划沉淀

    [加载阶段]
        第一步：依赖检测
            执行 [依赖检测]

        第二步：读取当前边界
            读取当前任务要求、根级入口和与当前任务直接相关的实现
            明确这次只动哪些文件 / 模块、哪些不动

        第三步：确认验证路线
            明确最小验证和必要回归验证怎么跑

    [实施阶段]
        第一步：评估影响范围
            先说明：
            - 当前主改动
            - 可能受影响的旧路径
            - 当前不会做的顺手项

        第二步：实施改动
            按当前边界完成最小必要改动

        第三步：补最小验证证据
            执行最小必要验证并记录真实结果

        第四步：指向下一步
            默认进入 `code-review`
            如验证暴露计划失效或风险变化 → 停下回 `dev-planner`

[工作流程（小任务轻计划实施模式）]
    触发条件：
    - 当前任务已有轻计划、边界摘要或明确下一步
    - 当前仍属 `Small` 路径，但值得先按轻计划实施

    [加载阶段]
        第一步：依赖检测
            执行 [依赖检测]

        第二步：读取轻计划输入
            读取当前会话中的轻计划、边界说明、关键文件 / 模块和最小验证路线
            检查这些输入是否仍与当前实现一致

        第三步：识别抬级信号
            检查是否已变成：
            - 范围扩大
            - 风险上升
            - 需要跨会话承接
            - 需要多阶段推进

            如已命中 → 暂停实施，回 `dev-planner`

    [实施阶段]
        第一步：按轻计划实施
            只实现当前轻计划明确覆盖的切片

        第二步：补证据
            运行最小验证和必要回归验证
            如本次已经接近中改或高风险，主动按 `Verification Evidence` 协议记录

        第三步：交给后续门禁
            默认进入 `code-review`
            命中发布、安全、AI 行为变更时，显式指向对应 `Gate`

[工作流程（正式计划实施模式）]
    触发条件：
    - 存在 `runtime/PLANS.md`
    - 或存在 `runtime/plans/active/*.md` 且当前任务已对应到某个活跃计划

    [加载阶段]
        第一步：依赖检测
            执行 [依赖检测]

        第二步：读取计划对象
            读取当前计划中的：
            - 任务目标
            - 当前状态
            - 范围与非目标
            - 关键文件 / 模块
            - 实施阶段拆解
            - 验证方式
            - 下一步

        第三步：确认当前阶段
            判断本次实施对应哪个阶段、哪个切片
            确认计划仍是最新，没有被验证结果或范围变化打断

        第四步：识别门禁
            根据当前阶段内容判断是否同步命中：
            - `code-review`
            - `release-builder`
            - `ADR-Lite`
            - `Security Lite`
            - `AI Eval Gate`

    [实施阶段]
        第一步：按计划对象实施
            不重做规划，只按当前阶段落代码和相关实现

        第二步：补统一证据
            中改及以上默认按 `Verification Evidence` 协议补证据

        第三步：判断是否继续当前计划
            - 计划仍有效 → 进入 `code-review` 或后续 `Gate`
            - 计划已失效 → 暂停实施，先回 `dev-planner` 更新计划对象

[工作流程（review 后 fix 回环模式）]
    触发条件：
    - 已有 `code-review` findings
    - 已有失败验证或回归场景
    - 当前任务本质是修正已有实现

    [加载阶段]
        第一步：依赖检测
            执行 [依赖检测]

        第二步：读取失败证据与 findings
            读取：
            - `runtime/code_review.md`
            - 相关 `runtime/reviews/`
            - 当前失败测试、复现步骤、回归场景、eval case

        第三步：锁定修正范围
            只针对当前 findings 或失败证据指向的问题实施，不顺手改其他方向

    [实施阶段]
        第一步：定向修正
            按 findings 或失败证据完成修正

        第二步：重新验证
            至少重新运行：
            - 原失败证据对应的验证
            - 修正后受影响路径的回归验证

        第三步：重新送审
            修后必须重新进入 `code-review`
            命中发布或其他门禁时，再叠加对应 `Gate`

[工作流程（阻塞与抬级）]
    触发条件：
    - 当前无法安全继续实施
    - 当前已命中其他门禁或计划更新条件

    [处理阶段]
        第一步：判断阻塞类型
            区分：
            - 需先补失败证据
            - 需先更新计划
            - 需进入 `release-builder`
            - 需进入 `Security Lite`、`ADR-Lite`、`AI Eval Gate`
            - 缺关键信息，无法继续

        第二步：停止盲目实施
            不用“先写一半再说”的方式绕过阻塞

        第三步：输出明确去向
            只在下面这些状态中选择一个作为当前结果：
            - `可直接实施`
            - `需先补失败证据`
            - `需先更新计划`
            - `已可送审`
            - `需进入 release / security / ai-eval gate`
