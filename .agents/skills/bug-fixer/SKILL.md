---
name: bug-fixer
description: 当已有明确问题、失败证据、回归缺陷或 `code-review` findings 需要定向修复时使用。它按 `V3` 的证据驱动方式定位根因、实施最小修复、补复验结果，并把结果送回 `code-review` 或命中的其他 `Gate`。
---

[任务]
    **直接缺陷修复模式**：用户直接报告 bug、异常或错误行为，且已有失败日志、复现步骤或报错信息时，先固定失败证据，再定位根因并实施最小修复。

    **验证失败回环模式**：当前任务在 `dev-builder` 的功能验证、回归验证或 AI 行为验证中失败时，读取失败证据，定向修复当前缺陷，不把回环修复扩成新一轮重构。

    **review findings 修复模式**：当前任务来自 `code-review` 的 findings、`Hold / Reject` 结论或 `runtime/reviews/` 中的明确问题时，只围绕这些问题修复，并在修后重新进入 `code-review`。

[调用上下文]
    `bug-fixer` 主要在 3 个场景被调用：
    1. 用户直接报告 bug → 主 Agent 调用 `bug-fixer` → 修后回到 `code-review`
    2. `dev-builder` 验证失败或回归失败 → 主 Agent 调用 `bug-fixer` → 修后重新补验证，再回到 `code-review`
    3. `code-review` 给出 findings 或 `Hold / Reject` → 主 Agent 调用 `bug-fixer` 修复明确缺陷 → 修后重新进入 `code-review`

[依赖检测]
    Skill 启动时第一步自动执行：

    必需：
    - 当前任务要求
    - 当前失败证据：报错信息、失败命令、复现步骤、回归场景或 `code-review` finding，至少有一种
    - 当前任务相关实现已存在
    - `README.md`、`AGENTS.md`、`PROJECT_RULES.md`
    - `docs/workflows/build-verify-review.md`

    按需必读：
    - `runtime/PLANS.md` → 当前任务已沉淀为正式计划时必须读取
    - `runtime/plans/active/*.md` → 多计划模式时必须读取对应计划对象
    - `runtime/code_review.md` → 当前修复来自 review 或修后要重新送审时必须读取
    - 当前任务相关的 `runtime/reviews/`、`runtime/errors/`、`runtime/releases/`、`runtime/adr/`
    - 当前任务直接相关的实现、测试入口和现有验证结果

    当前会话可复用：
    - 当前会话里已经形成的失败证据、轻计划边界、review findings 和验证结果
    - 但它们不是仓库真相源；只有未沉淀到 `runtime/` 时才作为当前输入使用

    可选（必要时核验）：
    - 外部依赖官方文档或正式资料 → 只在报错来源、接口规则或版本兼容性不确定时核验
    - 当前测试 / 构建入口 → 用于确认复验和回归验证怎么跑
    - `Product-Spec.md`、`Design-Brief.md`、设计稿 MCP、Playwright、git → 只有仓库本身明确需要时才作为辅助，不是默认前置

[第一性原则]
    **失败证据先于修复**：没有失败证据就不进入修复。至少要先固定报错信息、失败命令、复现步骤、回归场景或 review finding，再开始定位根因。

    **根因先于动作**：`bug-fixer` 不是“先改改试试看”。先说明根据哪些证据判断问题在哪一层、为什么是这个根因，再实施修复。

    **一次只修一个问题**：每轮只处理一个明确缺陷。相关联的必要辅助改动可以一起做，但不能借修 bug 顺手扩成目录重组、协议改写或无关重构。

    **修复也要守边界**：修 bug 不意味着可以跳过边界判断。动手前必须说明修复范围、受影响文件 / 模块和可能被带到的旧路径。

    **修后必须重新验证**：修完后至少重新运行：
    - 原失败证据对应的验证
    - 受影响路径的回归验证
    - 命中 AI 行为变更时的关键 case 或回归 case

    **修后必须回到 review**：`bug-fixer` 只负责定位和修复，不直接给出最终 `Accept / Hold / Reject`。修后默认重新进入 `code-review`。

    **计划失效先回 planner**：如果在修复过程中发现范围扩大、风险变化、验证结果改变路线或需要拆阶段推进，停止继续修复，先回 `dev-planner` 或更新 `runtime/PLANS.md`。

    **必要时核验，不靠过期记忆**：只有当第三方依赖报错、框架兼容性或外部规则不确定时，才查官方资料或正式来源，不把“联网搜索”扩成默认重型步骤。

    **门禁不能口头吞掉**：修复一旦命中 `Security Lite`、`ADR-Lite`、`AI Eval Gate` 或发布相关门禁，必须显式抬级，不把问题埋在修复说明里。

[输出风格]
    **语态**：
    - 像工程调试记录，先说证据，再说判断，再说修复和复验
    - 简洁、可复查、能直接交给下一位继续处理

    **原则**：
    - x 不说“我改了一下看看”
    - x 不在证据不足时硬下根因结论
    - x 不跳过复验就说“修好了”
    - x 不把最终收口结论冒充成 fix 结论
    - ✓ 先说明失败证据
    - ✓ 说明根因和修复范围
    - ✓ 给出重新验证结果和未解决风险
    - ✓ 明确下一步回到哪个 skill 或哪个 `Gate`

    **典型表达**：
    - "失败证据是 `npm test` 在 `session delete` 场景报错，根因定位到 `deleteSession` 没清理残留引用；本次只修 `session store` 和相关回归路径。"
    - "当前 findings 指向回归问题，修复后已重新执行原失败步骤和相关回归验证；下一步必须回到 `code-review`。"
    - "修复过程中发现问题已经从单点 bug 变成契约调整，当前停止继续修复，先回 `dev-planner` 更新计划对象。"

[调试规则清单]
    调试过程中必须遵守以下规则：

    [证据收集规则]
        - 失败证据至少包含一种：错误日志、失败命令、复现步骤、回归场景、`code-review` finding
        - 证据要能复查，不用“刚才报过一次”代替
        - 如来自正式计划或 review，先读取对应 `runtime/` 对象，再定修复范围

    [根因判断规则]
        - 每轮最多保留 `1-3` 个可验证假设，按可能性排序
        - 每个假设都要有验证方法
        - 假设被否定后说明原因，不重复兜圈子
        - 连续多轮修复都失败时，重新审视是不是计划失效、风险升级或问题并不在当前层

    [修复规则]
        - 一次只修当前明确缺陷
        - 修前评估影响范围
        - 修后至少补原失败验证 + 相关回归验证
        - 命中 AI 行为变更时补关键 case、回归 case 或 eval evidence

    [门禁抬级规则]
        - 触发 `security`、`data`、`public-api`、`infra` 或涉及外部数据 / 权限的 `ai-behavior` → 进入 `Security Lite`
        - 触发 Prompt、模型、RAG、tool routing、guardrail 变化 → 进入 `AI Eval Gate`
        - 触发跨模块结构调整、公共抽象、关键依赖替换、核心契约变化 → 进入 `ADR-Lite`
        - 修复影响交付或发布路线 → 同步标记后续 `release-builder`

    [状态分支]
        当前结果只允许落在下面几种状态之一：
        - `需先补失败证据`
        - `已定位可修`
        - `已完成待复验`
        - `需回 planner`
        - `需进 security / ai-eval gate`
        - `阻塞待确认`

[调试策略]
    **第一阶段：固定失败证据**
    - 读取报错信息、失败命令、复现步骤、回归场景或 findings
    - 判断当前问题属于用户直接报 bug、验证失败还是 review 回环
    - 如失败证据不足，先停在 `需先补失败证据`

    **第二阶段：定位根因**
    - 结合当前边界、计划对象和相关实现，判断问题在哪一层
    - 形成可验证假设并快速排除
    - 如果发现问题已超出当前计划或风险升级，进入 `需回 planner`

    **第三阶段：实施最小修复**
    - 只改当前缺陷需要的文件 / 模块
    - 不顺手扩写，不把修 bug 扩成重构
    - 如发现必须进入其他 `Gate`，先标出，再决定是否继续当前修复

    **第四阶段：复验并交回主链**
    - 重新运行失败证据对应的验证
    - 补受影响路径的回归验证
    - 输出根因、修复范围、复验结果、未解决风险和下一步
    - 默认回到 `code-review`

[工作流程]
    [启动阶段]
        第一步：依赖检测
            执行 [依赖检测]

        第二步：确定入口
            判断当前属于：
            - 用户直接报 bug
            - `dev-builder` 验证失败 / 回归失败
            - `code-review` findings 修复

        第三步：读取失败上下文
            读取当前失败证据、相关实现和按需运行对象
            如存在计划对象，同时确认当前计划仍有效

    [定位阶段]
        第一步：固定失败证据
            写清当前失败命令、复现步骤、回归场景或 findings

        第二步：定位根因
            基于证据形成可验证假设并判断最可能根因
            如证据不足 → 输出 `需先补失败证据`
            如问题已越过当前计划边界 → 输出 `需回 planner`

    [修复阶段]
        第一步：说明修复范围
            明确这次只修哪些文件 / 模块、可能影响哪些旧路径

        第二步：实施最小修复
            围绕当前根因完成最小必要改动

        第三步：识别门禁
            检查是否命中 `Security Lite`、`ADR-Lite`、`AI Eval Gate` 或发布链

    [复验阶段]
        第一步：运行原失败验证
            重新运行能证明“问题已修”的失败证据对应验证

        第二步：运行回归验证
            重新验证相关旧路径，防止修 A 坏 B

        第三步：输出 fix 结果
            输出：
            - `根因`
            - `修复范围`
            - `重新验证证据`
            - `未解决风险`
            - `下一步`

        第四步：指向下一步
            - 修复完成且已复验 → `已完成待复验`，默认进入 `code-review`
            - 需补 gate → `需进 security / ai-eval gate`
            - 证据不足或信息缺失 → `阻塞待确认`

[初始化]
    执行 [启动阶段]
